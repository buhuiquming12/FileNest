# FileNest 开发过程报告

> 桌面文件整理工具（JavaFX + AI 建议 · 个人版）
> 环境：Windows 11 · JDK 17.0.8（`D:\JDK`）· Maven 3.9.9 · JavaFX 17.0.13

---

## 摘要

本项目依据既定系统设计文档，从一个空的 Maven 骨架起步，实现了一个**单机、个人使用**的桌面文件整理工具。核心工程约束是一对矛盾——**AI 的不确定性 vs. 文件操作的确定性要求**——最终以「AI 只建议、人来确认、一切可撤销」的边界化解。

成果：**28 个主源文件 + 7 个测试类 / 20 个测试用例，`mvn compile` 与 `mvn test` 全部通过**。本报告记录需求边界、架构决策、实现要点，以及开发过程中真实遇到并解决的 4 个问题。

---

## 一、项目背景与目标

个人下载/桌面文件夹常年混乱：命名无意义（`新建文本文档(3).docx`）、扩展名不可靠、同一文档存了多个版本。目标是做一个**手动触发、可预览、可撤销**的整理工具，并用 AI 提供**建议**（而非自动决策）来处理规则难以覆盖的模糊场景。

硬约束：**不可丢文件、不可误删；所有「移动」必须可撤销。**

---

## 二、需求分析与边界澄清

开发前先对输入做了分拣，明确"做什么/不做什么"：

| 维度 | 决策 |
|---|---|
| 使用场景 | 单机个人，不考虑多用户/权限 |
| 触发方式 | 手动选文件夹整理，非后台常驻监控 |
| AI 定位 | **辅助建议**，输出方案，人确认后才真正移动 |
| 存储 | 本地文件系统 + 轻量 JSON 日志，不引入服务端/数据库 |
| 关键约束 | 不丢文件、不误删；所有移动可撤销 |

**刻意不做**（避免过度设计）：数据库/服务端、实时监控（`WatchService`）、让 AI 直接执行文件操作。

---

## 三、总体架构设计

采用严格分层，UI 只依赖应用层门面，核心模块互不知晓实现细节：

```
UI 层 (JavaFX)  →  OrganizeService 门面  →  Scanner / Classifier / AiAdvisor / FileExecutor / OperationLog
```

**最关键的结构决策**：`Classifier`（规则分类）与 `AiAdvisor`（AI 建议）是**并列而非串联**的两路独立信息源，由 `OrganizeService` 融合。这样即使 AI 离线/超时/异常，规则分类仍独立产出完整计划——**容错隔离**，AI 故障不会拖垮系统。

设计原则落地：SRP（六个模块各司其职）、OCP（`ClassifierStrategy`/`AiAdvisor` 接口化）、DIP（应用层只依赖接口）、关注点分离（UI 只认 `OrganizePlan`）、幂等可恢复（撤销=反向回放日志）。

---

## 四、核心设计决策与取舍

开发中做出并记录了以下关键决策（均与设计文档一致或为其自然延伸）：

1. **JDK/JavaFX 版本**：实测本机 JDK 为 17，故目标锁定 Java 17 + JavaFX 17 LTS。
2. **UI 用纯代码而非 FXML**：单人小工具，纯代码构建减少 FXML 加载/控制器装配的活动部件。
3. **AI 默认走本地启发式**：`HeuristicAiAdvisor` 离线可跑、无需 API Key，开箱即用；同时保留 `LlmAiAdvisor` 接入骨架与 `NoOpAiAdvisor` 降级实现，接口不变即可替换。
4. **精确用算法、模糊才用 AI**：逐字节重复用 SHA-256（比任何模型都可靠），AI 只负责"相似但不相同"的模糊判断——用对工具，不滥用 AI。
5. **安全不变量写进类型系统**：`FileAction` 构造器强制"AI 来源 ⇒ 必须确认"，从根上杜绝 AI 动作被漏标为"无需确认"。
6. **冲突绝不覆盖**：`FileExecutor` 只做「跳过/自动重命名」，永不 `REPLACE_EXISTING`。
7. **撤销基于事实而非猜测**：`OperationLog` 记录每次「源→目标」，撤销=反向 `move`。

## 五、核心实现要点

### 5.1 数据契约 `FileAction`
全系统的统一语言（record），UI/执行器/日志共用。构造器内置安全校验：`confidence` 越界抛异常；`source==AI` 时强制 `requiresConfirm=true`。提供 `rule(...)`/`ai(...)`/`skip(...)` 工厂，使不变量无法被绕过。

### 5.2 规则分类与去重（确定性层）
- `ExtensionClassifier`：按扩展名映射到 `文档/图片/视频/…`（`CategoryRules` 维护映射），或按修改月份归档；置信度恒为 1.0、无需确认。
- `DuplicateDetector`：先按**文件大小预筛**（同尺寸才可能相同），再对候选算 SHA-256，避免无谓哈希。

### 5.3 本地启发式 AI（`HeuristicAiAdvisor`）
覆盖设计的三个 AI 功能，全部为"建议"：
- **内容级分类**：正则识别 `截图/IMG_/微信图片/发票/简历/合同/报告` 等命名；对 `txt/无扩展名` 文件**嗅探前 2KB** 内容，识别 CSV / JSON / 源代码，纠正误导性扩展名。
- **相似检测**：`normalizeStem` 归一化文件名词干（剥离 `副本/最终版/v1/(1)/数字` 等噪声）后聚类，提示疑似同组。
- **结构建议**：按项目模式下把同组文件建议归入 `项目_xxx`。

### 5.4 超时降级（`TimeoutAiAdvisor` 装饰器）
在独立守护线程中调用被装饰的 advisor，`Future.get(timeout)` 超时则取消并回退到 `NoOpAiAdvisor`；捕获一切异常同样回退。这实现了设计中"AI 超时/不可用 → 降级、规则结果照常展示"的要求。

### 5.5 执行与撤销（`FileExecutor` + `JsonOperationLog`）
- 执行：逐条移动，先 `createDirectories` 建目标目录；目标存在则按策略跳过或生成 `name (1).ext`；成功即记录 `OperationRecord`。整批完成写入一个 `OperationBatch`。
- 撤销：**逆序**回放，把 `to→from` 移回；撤销后**尽力清理空目录**。
- 持久化：`JsonOperationLog` 用 Jackson，把 `Path` 存为字符串 DTO（可读、可移植），默认写 `~/.filenest/operations.json`；读损坏文件不崩溃、以空日志启动。

### 5.6 UI（`MainView`）
目录选择 → 计划表格（`文件/建议去向/依据/来源/置信度`，置信度按 绿/橙/红 分级）→ 勾选 → 执行/撤销。**低置信度、AI 与重复项默认不勾选**；扫描/执行/撤销均在 JavaFX `Task` 后台线程执行，避免卡界面；执行前有二次确认对话框。

## 六、开发过程与问题排查（踩坑记录）

开发按依赖顺序推进：**构建配置 → 数据模型 → 核心模块 → 应用门面 → UI → 测试**。过程中先编译、后测试，逐步发现并修复了 3 个问题。

### 问题 1：构建目标 JDK 与实机不匹配（编译失败）
- **现象**：初始 `pom.xml` 设 `maven.compiler.source/target=19`，但实测本机 `D:\JDK` 为 **JDK 17.0.8**。
- **原因**：目标字节码版本高于实际编译器版本，无法编译。
- **修复**：改为 Java 17，并补齐 JavaFX 17.0.13 / Jackson 2.17.2 / JUnit 5.10.2 依赖与 `javafx-maven-plugin`；`groupId` 由模板遗留的 `com.jobradar` 规范为 `com.filenest`。

### 问题 2：record 访问器被误写成字段访问（编译失败）
- **现象**：`FileExecutor.java:49` 报 `sourcePath in FileAction has private access`。
- **原因**：写成了 `action.sourcePath`（访问 record 的 private 字段），应为 `action.sourcePath()`（record 生成的 public 访问器方法）。
- **修复**：补上括号。重新编译通过（`EXIT=0`）。

### 问题 3：中文场景下正则 `\b` 词边界失效（测试失败）★
这是最有价值的一个坑，值得展开。

- **现象**：`normalizeStem("报告v1")` 期望剥离版本号 `v1` 得到 `报告`，实际得到 `报告v`——测试 `报告v1` 与 `报告最终版` 归一化后应相等，却不相等而失败。
- **定位**：用最小 Java 程序实证对比，结论清晰：

  | 输入 | `\bv\d+\b` 是否匹配 |
  |---|---|
  | `ab v1`（空格+v） | ✅ 匹配 |
  | `报告 v1`（空格+v） | ✅ 匹配 |
  | `报告v1`（汉字紧邻 v） | ❌ **不匹配** |
  | `abcv1`（字母紧邻 v） | ❌ 不匹配 |

- **根因（与直觉相反）**：**Java 正则把 CJK 汉字视为「词字符（word character）」**。因此 `告` 与 `v` 两侧都是词字符，之间**不构成词边界**——就像 `abcv1` 里 `c` 与 `v` 之间没有边界一样，`\bv` 自然匹配失败。（加 `UNICODE_CHARACTER_CLASS` 亦然，汉字仍是词字符。）我最初"汉字是非词字符、应有边界"的假设是错的，实证纠正了它。
- **修复**：改用**负向后行断言** `(?<![a-z0-9])v\d+`——只要求 `v` 前不是 ASCII 字母/数字。汉字、空格、行首都满足（能剥离 `报告v1`），而 `cv2` 这类词内 `v` 不会被误删。修复后该用例及全部 20 个测试通过。

### 问题 4：运行时报「缺少 JavaFX 运行时组件」（IDE / 直接启动）
- **现象**：从 IDE 直接运行 `App`，或 `java -cp ... com.filenest.App` 时报
  `错误: 缺少 JavaFX 运行时组件, 需要使用该组件来运行此应用程序`，退出码 1。
- **原因**：JDK 启动器有一项检查——当**主类继承 `javafx.application.Application`** 且
  `javafx.graphics` 不在 **module path** 上时，即拒绝启动。IDE 把 JavaFX 作为普通依赖放在
  **classpath**（unnamed module），故触发该检查。（`mvn javafx:run` 因插件已配置 module path 而不受影响。）
- **修复**：新增**不继承 `Application`** 的入口类 `Launcher`，由它调用 `App.main(...)`；主类不继承
  `Application` 即绕过该检查，JavaFX 从 classpath 正常加载。并将 Maven 插件的 `mainClass` 统一为 `com.filenest.Launcher`。
- **验证**：本机 A/B 实证——旧入口 `App` 复现该错误并退出 1；新入口 `Launcher` 成功
  `PlatformImpl startup`（JavaFX 启动、窗口拉起），仅剩一行无害的 unnamed-module 警告。

## 七、测试与验证

### 测试策略
不测 GUI 像素，而是对**安全关键的纯逻辑**做端到端与单元测试，用 JUnit 5 `@TempDir` 在临时目录里真实地建文件、移文件、撤销。

| 测试类 | 用例数 | 覆盖 |
|---|---|---|
| `ExtensionClassifierTest` | 3 | 已知/未知扩展名、按日期归档 |
| `DuplicateDetectorTest` | 2 | 相同内容成组、不同内容不成组 |
| `FileExecutorTest` | 3 | 移动+日志+撤销回放、冲突不覆盖、跳过策略 |
| `HeuristicAiAdvisorTest` | 4 | 截图识别必须确认、CSV 内容嗅探、词干归一化、干净文件不打扰 |
| `TimeoutAiAdvisorTest` | 3 | 超时降级、异常降级、正常透传 |
| `JsonOperationLogTest` | 2 | 持久化跨实例重载、删除持久化 |
| `OrganizeServiceTest` | 3 | 计划只读且 AI 全需确认、重复检测标记、执行→撤销往返 |
| **合计** | **20** | |

### 验证结果（据实说明）
- ✅ `mvn -DskipTests compile`：通过（含 UI 层）。
- ✅ `mvn test`：**Tests run: 20, Failures: 0, Errors: 0**，`BUILD SUCCESS`。
- ⚠️ **GUI 未在本开发环境做可视验证**：无桌面显示，未启动窗口；但 UI 是应用层门面之上的薄层，其真实行为（扫描→计划→执行真实移动→撤销）已由 `OrganizeServiceTest` 端到端覆盖。GUI 需在有桌面的机器上 `mvn javafx:run` 亲验。

> 说明：测试日志里出现的两行 `[AI] ... timed out / failed ... falling back` 是**超时/异常降级用例的预期输出**，非报错。

## 九、局限与后续可扩展

- **接入真实大模型**：实现 `LlmAiAdvisor.callModel()`（`java.net.http.HttpClient` + 从 `FILENEST_LLM_ENDPOINT`/`FILENEST_LLM_API_KEY` 读配置）即可替换默认启发式，UI/执行/撤销零改动。
- **递归子目录整理**：当前仅扫描顶层常规文件（更安全）；可扩展为可选递归。
- **更强的相似检测**：现为文件名词干聚类，可加入内容级近似（如 SimHash / 文本嵌入）。
- **持久化升级**：JSON 日志足够个人使用；若需检索历史，可平滑替换为 SQLite（`OperationLog` 已接口化）。

## 十、总结

项目在既定设计边界内完整落地：分层清晰、AI 可插拔、安全可撤销，20 个测试守住了"不丢文件/不覆盖/可回退/AI 必确认"的关键路径。开发过程中四个真实问题（JDK 版本不匹配、record 访问器误用、CJK 词边界、JavaFX 运行时组件缺失）均已定位根因并修复，其中"Java 视汉字为词字符导致 `\b` 失效"一条经最小程序实证，是本次开发最有价值的技术沉淀。
