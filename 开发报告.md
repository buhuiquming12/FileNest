# FileNest 开发过程报告

> 桌面文件整理与空间分析工具（JavaFX + AI 建议 · 个人版）
> 最新更新：2026-07-22
> 环境：Windows 11 · JDK 17.0.8（`D:\JDK`）· Maven 3.9.9 · JavaFX 17.0.13

---

## 摘要

FileNest 是一个单机桌面文件整理工具。本轮在原有“规则分类、AI 建议、执行预览、操作撤销”能力上，完成了空间占用分析、全部文件清单、独立清理建议、URL 模型发现以及安全自动整理等功能。

当前系统将**扫描、整理建议、清理建议、执行**拆成相互独立的步骤：扫描只收集文件系统事实，不调用 AI；整理建议和清理建议按需生成；真正移动文件前必须经过客户端安全策略。AI 可以参与建议和自动整理候选筛选，但不能绕过本地校验直接操作文件系统。清理建议本身只读，不自动删除文件。

最新成果：**49 个主源文件、14 个测试类、36 个测试用例**。最终执行 `mvn package`，结果为 **Tests run: 36, Failures: 0, Errors: 0, Skipped: 0，BUILD SUCCESS**，构建产物为 `target/FileNest-1.0-SNAPSHOT.jar`。

## 一、项目背景与最新目标

个人下载目录、桌面和资料目录常见以下问题：

- 文件类型多、命名混乱，未知扩展名和无扩展名文件容易被遗漏；
- 不清楚空间主要被哪些子文件夹或大文件占用；
- 规则可以完成确定性分类，但难以理解文件名、内容和项目语义；
- 自动移动或删除一旦缺少边界检查，可能造成覆盖、越界和误操作。

因此项目的最新目标是：

1. 扫描后展示当前目录的**全部常规文件**，不因扩展名未知而隐藏；
2. 展示各直接子文件夹的递归占用、占比、文件数、子目录数及大文件；
3. 将扫描与 AI 建议彻底分离，使 AI 不可用时仍可完成扫描；
4. 支持本地启发式建议和 URL 大模型，并可从 URL 获取模型列表；
5. 提供规则安全自动整理和 AI 建议自动整理；
6. 用白名单、路径约束、置信度门槛、二次确认、日志和撤销保证安全。

核心约束保持不变：**不覆盖原文件、不允许路径越界、移动操作可撤销、删除操作必须限定在明确白名单内。**

## 二、需求边界与产品决策

| 维度 | 当前决策 |
|---|---|
| 使用场景 | 单机个人使用，不引入多用户系统或服务端数据库 |
| 触发方式 | 用户手动选择目录，不做后台常驻监控 |
| 普通整理扫描 | 只读取所选目录顶层文件，不递归搬动已有子目录内容 |
| 文件展示 | 展示已知、未知、无扩展名、隐藏及系统配置文件；危险项只展示、不自动整理 |
| 空间分析 | 递归统计所选目录，但不跟随符号链接、不修改文件 |
| AI 定位 | 生成整理或清理建议；自动整理也必须通过本地安全门和人工确认短语 |
| 清理建议 | 独立生成，只读展示，不提供“按 AI 建议直接删除” |
| C 盘清理 | 仅处理预定义白名单位置中本次扫描返回且人工勾选的项目 |
| 持久化 | 使用本地 JSON 操作日志，不引入数据库 |
| 冲突处理 | 仅跳过或自动改名，绝不覆盖同名文件 |

项目仍刻意不做：后台实时监控、让模型直接访问文件系统、根据 AI 文本无校验地执行命令、全盘无边界自动删除。

## 三、总体架构

系统继续采用分层设计，并将“事实收集”和“建议生成”进一步拆开：

```text
JavaFX UI
├─ OrganizeService
│  ├─ scan()       → FileScanner                         （事实）
│  ├─ suggest()    → Classifier / Duplicate / AiAdvisor （建议）
│  └─ execute()    → Safety checks / FileExecutor / Log （执行与撤销）
├─ StorageAnalysisService
│  ├─ scan()       → FolderSizeService                  （空间事实）
│  ├─ advise()     → CleanupAdvisor                     （清理建议）
│  └─ fetchModels()→ UrlModelCatalog                    （URL 模型发现）
└─ DiskCleanupService                                   （白名单清理）
```

关键设计点：

- `OrganizeService.scan(...)` 返回 `OrganizeScan`，`suggest(...)` 消费已有扫描结果，扫描时不会调用 AI；
- `StorageAnalysisService.scan(...)` 只递归统计空间，`advise(...)` 才调用本地或远程清理顾问；
- 规则分类与 AI 建议仍是并列信息源，AI 超时或失败不会影响规则结果；
- 模型只返回结构化建议，客户端负责文件名、目标路径、来源、置信度和执行范围校验；
- `FileExecutor` 与 `AutoOrganizeSafetyPolicy` 分别在候选筛选和实际执行阶段防御风险。

这种结构符合 SRP、OCP 和 DIP：扫描器不判断、顾问不执行、执行器不信任上游输入、UI 只负责展示与确认。

## 四、本轮功能实现

### 4.1 扫描与建议分离

原有一体式 `plan(...)` 被拆为：

- `scan(OrganizeContext)`：读取文件元数据并完成确定性扫描；
- `suggest(OrganizeScan, OrganizeContext)`：基于已有扫描生成规则、重复文件和 AI 建议；
- `plan(...)`：保留为兼容入口，内部组合前两个步骤。

UI 中扫描和生成建议使用不同按钮。用户可以先查看完整文件事实，再决定是否调用 AI，避免网络请求阻塞或污染扫描流程。

### 4.2 全部文件清单与类型扩充

`FileScanner` 读取所选目录顶层的全部常规文件，仅跳过 FileNest 自身的 `.filenest` 元数据目录；单个文件不可读时记录提示并继续扫描，不让整个任务失败。

“全部文件（当前目录）”页面显示：

- 文件名、扩展类型与归类；
- 文件大小、修改时间；
- 是否隐藏、是否为系统配置、是否为符号链接；
- 是否允许参与自动整理的安全状态。

`CategoryRules` 已扩展到文档、表格、演示、图片、音频、视频、压缩包、程序、代码、电子书、字体、数据、设计与三维等类别。未知扩展名和无扩展名文件仍会出现在清单中，只是归入“其他”且不进入规则自动整理。

### 4.3 文件夹占用与大文件分析

`FolderSizeService` 使用只读递归遍历，一次扫描生成：

- 所选目录总大小、总文件数、子文件夹数和不可访问项数；
- 每个直接子文件夹的递归占用、占比、文件数和子目录数；
- 根目录直接文件的汇总；
- 最多 100 个大文件，按大小倒序展示。

遍历不跟随符号链接，遇到不可访问文件时累计计数并继续，不修改任何文件。

### 4.4 独立 AI 清理建议

新增 `CleanupAdvisor` 抽象及两种实现：

- `HeuristicCleanupAdvisor`：离线识别缓存、临时文件、旧安装包等候选；
- `LlmCleanupAdvisor`：向用户配置的 URL 模型请求结构化建议。

清理建议只消费已经完成的 `StorageScanResult`，不会重新遍历磁盘。结果按“可删除 / 建议检查 / 建议保留”展示路径、大小、理由、置信度和来源。远程 AI 异常时自动回退到本地建议；该页面没有直接删除入口。

### 4.5 URL 模型获取与远程 AI 配置

主界面支持填写 AI API 地址、API Key 和模型名。`UrlModelCatalog` 可以根据聊天端点推导并请求模型列表：

- OpenAI 兼容接口：`/models`；
- Ollama 接口：`/api/tags`。

返回模型会填充到可编辑下拉框。应用配置后，整理建议和清理建议分别使用远程模型；地址留空可切回本地启发式模式。HTTP 端点、响应 JSON、建议文件名、相对目标路径和置信度均由客户端校验。

### 4.6 两种安全自动整理

新增 `AutoOrganizeSafetyPolicy`，提供两种模式：

1. **规则安全自动整理**：仅选择来源为规则、扩展名已知、无需确认且通过安全校验的动作；未知类型和“其他”不会自动移动。
2. **AI 建议自动整理**：仅选择 AI 来源、置信度不低于 `0.80` 且通过安全校验的动作；执行前必须输入短语 `确认AI整理`。

自动模式单批最多处理 500 项，只处理所选目录顶层源文件，目标必须仍在该目录中，并保持原文件名。被拒绝的动作只计数和提示，不会偷偷降级执行。

### 4.7 C 盘安全清理

`DiskCleanupService` 提供与 AI 清理建议分开的确定性清理功能，白名单包括用户临时目录、Windows 临时目录、应用崩溃转储和 Windows 更新下载缓存。

流程为“扫描预览 → 用户勾选 → 二次确认 → 删除”。最近 24 小时项目默认不选；服务只接受本次白名单扫描返回的子项，不跟随符号链接、不删除白名单根目录，并拒绝伪造或越界候选。

## 五、安全设计

安全性不是只依赖 UI，而是贯穿数据契约、候选筛选、执行和撤销四层。

### 5.1 类型层不变量

`FileAction` 构造器校验置信度范围，并强制所有 `source == AI` 的动作都设置 `requiresConfirm = true`。因此 AI 动作不可能被上游误标为普通规则动作。

### 5.2 自动整理安全门

`AutoOrganizeSafetyPolicy` 拒绝以下情况：

- 源文件不在所选目录顶层，或目标越出所选目录；
- 目标层级异常、目标文件名与源文件名不一致；
- 根目录是磁盘根、用户主目录或系统关键目录；
- 源是隐藏文件、系统配置文件、点文件或符号链接；
- 路径涉及 `.filenest` 或隐藏目标目录；
- 规则模式遇到未知扩展名、“其他”类别或需确认动作；
- AI 模式置信度低于 80%；
- 批量动作超过 500 项。

### 5.3 执行器二次校验

`FileExecutor` 不假设上游可信。执行和撤销前再次检查标准化路径、根目录边界、符号链接和保护目录；目标冲突时只执行 `SKIP` 或生成 `name (1).ext`，从不使用 `REPLACE_EXISTING`。

### 5.4 日志与撤销

每次成功移动都会记录 `OperationRecord`，整批形成 `OperationBatch` 并持久化到 `~/.filenest/operations.json`。撤销按逆序把 `to → from` 回放，并尽力清理空目录。撤销遇到冲突同样遵守不覆盖策略。

### 5.5 AI 最小权限

AI 只接收用于生成建议的受控元数据或文本片段，不获得 Java 文件操作对象，也不能执行命令。远程返回值必须转换为本地 `FileAction` 或 `CleanupSuggestion` 后才能展示；真正移动仍由本地代码决定，AI 清理建议则始终只读。

## 七、开发过程与问题排查

开发顺序为：构建配置 → 数据模型 → 核心模块 → 应用门面 → UI → 安全策略 → 测试。历史上定位并修复了以下关键问题：

### 7.1 JDK 目标版本不匹配

初始模板使用 Java 19，而实际环境为 JDK 17.0.8，导致编译失败。项目最终统一到 Java 17，并配套 JavaFX 17.0.13、Jackson 2.17.x、JUnit 5.10.x 和 JavaFX Maven 插件。

### 7.2 record 访问器误用

曾将 `FileAction` 的 record 组件写成字段访问 `action.sourcePath`，编译器提示 private access。修复为 `action.sourcePath()` 后通过编译。

### 7.3 中文文件名中的正则词边界


ormalizeStem("报告v1")` 未能正确剥离版本号。原因是 Java 正则将 CJK 汉字视作词字符，`报告v1` 中汉字与 `v` 之间不存在 `\b` 边界。最终改用 `(?<![a-z0-9])v\d+`，既支持中文紧邻版本号，也避免误删英文单词内部的 `v`。

### 7.4 JavaFX 直接启动缺少运行时组件

直接运行继承 `Application` 的 `App` 会触发 JDK 对 module path 的检查。新增不继承 `Application` 的 `Launcher`，由其调用 `App.main(...)`，并将 Maven 主类统一为 `com.filenest.Launcher`。

### 7.5 功能扩展后的安全边界调整

“AI 只建议”扩展为“用户可确认 AI 自动整理”后，不能只依赖 UI 勾选状态。为此新增独立的 `AutoOrganizeSafetyPolicy`，并在 `FileExecutor` 中加入执行阶段复核；同时把清理建议和实际删除分成两个模块，避免“模型说可删”直接变成删除动作。

## 十、已知边界与后续方向

- 普通整理仍只自动处理所选目录顶层文件，这是有意采用的保守默认值；
- 未知、隐藏、系统配置和符号链接文件可以显示，但不会进入自动整理；
- AI 清理建议不会自动删除，用户若要清理必须使用独立的白名单清理流程或自行核对；
- 远程模型需兼容当前 JSON 契约，异常或格式错误时会回退或丢弃不安全建议；
- 可继续扩展内容级近似检测、更多模型协议、操作历史 UI 和可选的递归整理预览，但任何扩展仍需保持路径约束和可撤销原则。

## 十一、总结

FileNest 已从基础文件分类工具升级为“**文件清单 + 空间分析 + 独立 AI 建议 + 安全自动整理 + 可撤销执行**”的一体化桌面工具。系统能够显示所有常规文件类型及未知文件，分析不同文件夹的占用，按需生成整理和清理建议，从 URL 获取模型，并依据规则或高置信度 AI 建议自动整理。

本轮最重要的改进不是简单增加自动化，而是为自动化补齐边界：扫描与建议分离、AI 与执行分离、候选筛选与执行复核分层、清理建议与删除分离。配合 36 个测试、无覆盖策略、路径约束、确认短语、日志和撤销机制，项目在提升效率的同时保持了对用户文件的保守处理。
